# 架构设计

本文描述当前代码的实际结构。

## 1. 分层

系统可以分为四层：

1. 接入层
   `backend/main.py`、`backend/websocket_handler.py`、`ui/`
2. 编排层
   `core/agent.py`、`agents/`
3. 能力层
   `skills/`、`tools/`
4. 基础设施层
   `infra/`、`core/memory.py`、`core/critic.py`、`core/reflect.py`

## 2. 主执行链路

Web 入口链路：

```text
frontend/src/websocket.js
  -> ws://localhost:8000/pubsub
  -> backend/websocket_handler.dispatcher()
  -> backend.session.sessions.get_or_create()
  -> core.agent.Agent.handle()
  -> ManagerAgent.plan()
  -> LearningAgent.execute_dag()
  -> OrchestratorAgent.stream()/orchestrate()
  -> push event back to frontend
```

同步入口链路：

```text
ui/cli.py or ui/app.py
  -> Agent.chat()
  -> Agent.handle()
```

## 3. 会话模型

会话由 `backend/session.py` 管理。

- 键是 `client_id`
- 值是 `Session`
- 每个 `Session` 持有一个独立 `Agent`
- `Session` 带 `created_at`、`last_active`
- 支持 `dispose_callbacks`
- `dispose_callbacks` 现在支持异步回收

当前特征：

- 浏览器端 `client_id` 存在 `localStorage`
- 后端会按 `client_id` 复用上下文
- GC 每 60 秒执行一次
- 超过 `session_ttl_s` 的会话会被清理

## 4. Agent 聚合关系

`core/agent.py` 是运行时总入口，内部组合了：

- `ManagerAgent`
- `LearningAgent`
- `OrchestratorAgent`
- `DAGExecutor`
- `SkillTrainer`
- `ExecutionCritic`
- `Context`

职责分工：

- `ManagerAgent` 决定是不是闲聊、教学、重试或工具型任务
- `LearningAgent` 负责工具调用和依赖调度
- `OrchestratorAgent` 根据工具结果生成回复
- `SkillTrainer` 处理显式教学
- `ExecutionCritic` 在主回复完成后异步评估执行质量

## 5. 通信模型

当前后端不是直接手写裸 WebSocket 协议，而是基于 `fastapi-websocket-pubsub`。

主题约定：

- `init/{client_id}`
- `chat/{client_id}`
- `reset/{client_id}`
- `ping/{client_id}`
- `events/{client_id}`
- `log/{client_id}`

事件输出通常包括：

- `connected`
- `thinking`
- `plan`
- `tool_call`
- `tool_result`
- `tool_error`
- `message_delta`
- `message_final`
- `skill_learned`
- `teaching_interactive`
- `reset_ack`
- `pong`
- `error`

## 6. 启动与关闭

启动逻辑在 `backend/main.py`。

启动阶段：

1. `config.validate()`，配置错误直接抛出
2. 注册 PubSub dispatcher
3. 初始化 LLM provider registry
4. 启动 session GC 后台任务
5. 初始化 `ToolHub`
6. 连接工具源
7. 如果开启自演化，则启动 `SelfReflectLoop`

关闭阶段：

1. 停止 GC 任务
2. 停止反思循环
3. 调用 `ToolHub.disconnect_all()` 关闭工具资源

## 7. 设计取舍

当前架构的取舍比较明确：

- 保留同步 `Agent.chat()`，避免 CLI / Streamlit 入口全部重写
- `SessionManager` 目前仍是内存实现，简单但不支持多实例共享
- 工具层以 `ToolHub` 为中心，减少旧版多注册表并存的问题
- 自演化先走“记录 + 审批 + 持久化 skill YAML”，不直接改核心代码

## 8. 当前限制

- 会话存储没有外部持久化，服务重启后上下文会丢失
- `ExecutionCritic` 的 `fallback`、`retry` 统计仍不完整
- 工具源初始化失败目前只记录 warning，不是强 fail-fast
- 部分源码文件有编码问题，影响维护效率
