# 01 — 总体架构设计

> 本文档描述改进后的**目标架构**,先讲问题,再讲方案,最后给出分层与契约。

---

## 1. 现状问题清单

来自对当前代码的逐文件阅读,主要问题按"影响范围"从大到小排列:

| # | 问题 | 影响 |
|---|---|---|
| P1 | 三个 Agent 共享 LLM 调用逻辑,但没有 `BaseAgent` 抽象,职责重叠 | 任何 Agent 改动都得复制粘贴 |
| P2 | Manager 输出 JSON 解析靠正则,**无重试、无 Schema 校验** | LLM 输出稍变就整条请求失败 |
| P3 | 技能(Skill)只把 `method/steps` 拼到 prompt,**Orchestrator 是否遵循完全靠 LLM 自律** | 行为不可控、不可测 |
| P4 | 技能声明了 `code` 字段,**运行时从不读取** | 与 README 描述不符 |
| P5 | `execute_tasks` 始终串行,**`ThreadPoolExecutor` 创建了从不使用** | 性能浪费,无法表达并行 |
| P6 | **没有工具依赖图**:Manager 无法表达"先查天气再决定搜什么" | 复杂任务无法建模 |
| P7 | 工具失败/数据不足,**无法重试或追加调用** | 一次失败 = 整次失败 |
| P8 | Orchestrator 拿到的是拼接字符串,**结构化数据丢失** | 无法缓存、无法二次推理 |
| P9 | **多轮对话 Context 完全没用**(`self.context = []` 从未被赋值) | 用户问"那个景点"Agent 不懂 |
| P10 | **用户"教导技能"未实现**(README 承诺,代码无) | 核心卖点缺失 |
| P11 | WebSocket 日志订阅 `try` 加 `finally` 才 unsubscribe | 长时间运行会内存泄漏 + 重复推送 |
| P12 | `client_id = str(id(websocket))` | 进程重启即失效,无断线恢复 |
| P13 | 前端只显示文本,**流转可视化缺失** | 卖点"看到思考"无法兑现 |
| P14 | 无 `.env.example`、无测试、无 Docker | 不可部署、不可回归 |

## 2. 设计原则

1. **单一职责**:每个 Agent 只做一件事,LLM 调用通过 `BaseAgent.think()` 统一
2. **技能可执行**:技能不能只是 prompt 拼字符串,必须是 **可解析的执行图**
3. **可观测优先**:所有跨模块调用都要进 Trace,前端能看到完整链路
4. **失败可恢复**:LLM 输出错就重试,工具失败可重试/降级,网络断了能续上
5. **向后兼容**:改进时分阶段,**保留旧接口**,通过 feature flag 切换

## 3. 目标架构

### 3.1 分层

```
┌────────────────────────────────────────────────────────────────┐
│  Layer 4   UI / API      │ Vue 3 + WebSocket │ Streamlit │ CLI │
├────────────────────────────────────────────────────────────────┤
│  Layer 3   Orchestration │ core/agent.py (串联 + DAG 执行)     │
├────────────────────────────────────────────────────────────────┤
│  Layer 2   Agents        │ Manager / Learning / Orchestrator   │
│                          │ (继承 BaseAgent)                    │
├────────────────────────────────────────────────────────────────┤
│  Layer 1   Domain        │ Skills / Tools / Capabilities       │
├────────────────────────────────────────────────────────────────┤
│  Layer 0   Infra         │ LLMClient / Logger / Config         │
└────────────────────────────────────────────────────────────────┘
```

### 3.2 模块依赖

```
        ┌──────────┐
        │  UI/API  │  只依赖 core + infra
        └────┬─────┘
             │
        ┌────▼─────┐
        │  Core    │  串联 Agent,管理 Context,持有 Capability/Plugin 注册表
        └────┬─────┘
             │
   ┌─────────┼──────────┐
   │         │          │
┌──▼──┐  ┌───▼───┐  ┌───▼────────┐
│Agents│  │Skills │  │   Tools    │
└──┬──┘  └───┬───┘  └───┬────────┘
   │         │          │
   └─────────┴──────────┴────► Infra (LLM / Logger / Config)
```

**关键约束**:
- Agents 不直接 import 其他 Agent(避免循环),通过 `core` 编排
- Skills/Tools 不依赖 Agents
- Infra 是最底层,被所有上层引用,但不引用任何上层

### 3.3 请求处理流水线(目标态)

```
[WebSocket msg] → backend/main.py
       │
       ▼
[core/agent.py: Agent.handle(message, context)]
       │
       │ start_trace("chat-...")
       ▼
┌──────────────────┐
│  ManagerAgent    │  think(plan_schema) → {intent, skill_ref, dag}
└────────┬─────────┘            ▲
         │                      │ retry if invalid JSON (max 3)
         ▼
┌──────────────────┐
│  SkillRegistry   │  match(dag) → Skill(spec_version, steps[])
└────────┬─────────┘
         │
         ▼
┌──────────────────────────────────────────────┐
│  DAG Executor (core/agent.py 内)             │
│  按拓扑序执行 steps,每个 step:                │
│    1. resolve inputs from previous outputs    │
│    2. dispatch to LearningAgent               │
│    3. validate against step.output_schema     │
│    4. on failure: retry / fallback step       │
└────────┬─────────────────────────────────────┘
         │ per-step outputs
         ▼
┌──────────────────┐
│ Orchestrator     │  per-step: fill template / refine
│  (per step LLM)  │  final:   generate response
└────────┬─────────┘
         │
         ▼
[WebSocket msg stream] → frontend
       │
       │ end_trace()
       ▼
[LogEntry stream] → frontend (可视化)
```

## 4. 核心抽象

### 4.1 `BaseAgent`(新增,放 core/agent_base.py)

```python
class BaseAgent(ABC):
    name: str

    def __init__(self):
        self.llm = get_llm_client()
        self.logger = get_logger()

    @abstractmethod
    def system_prompt(self) -> str: ...

    def think(
        self,
        user_prompt: str,
        output_schema: Optional[dict] = None,  # JSON Schema
        retries: int = 3,
    ) -> Any:
        """统一 LLM 调用入口:
        1. 拼 messages
        2. 请求 LLM
        3. 若有 output_schema:解析 + 校验,失败重试
        4. 记录 LLM_REQUEST / LLM_RESPONSE 日志
        """
        ...

    def think_json(self, user_prompt: str, schema: dict) -> dict:
        """think 的便捷版,返回 dict"""
        ...
```

收益:
- Manager / Orchestrator 复用同一套解析/重试
- 所有 LLM 调用都带 trace_id,日志能串联

### 4.2 `Skill` 升级为 DAG(改 skills/manager.py)

现状 `Skill` 是 dataclass with `method/steps/code`。升级后:

```python
@dataclass
class SkillStep:
    id: str                       # 步骤唯一 id
    name: str
    description: str              # 给 LLM 看的说明
    tool: Optional[str] = None    # 关联工具名
    input_schema:  dict = field(default_factory=dict)   # JSON Schema
    output_schema: dict = field(default_factory=dict)
    depends_on:    List[str] = field(default_factory=list)  # 上游 step id
    template:      Optional[str] = None  # 输出模板
    fallback:      Optional[str] = None  # 失败时的降级 step id


@dataclass
class Skill:
    name: str
    version: str                  # 语义化版本
    method: str                   # 总方法论(给 Orchestrator 看)
    capability: str               # 能力描述(用于匹配)
    patterns: List[str]           # 意图匹配的关键词/正则
    tags: List[str]
    steps: List[SkillStep]
    created_at: str
    updated_at: str
    source: str                   # "builtin" / "taught" / "imported"
    author: Optional[str] = None
```

加载器在 `skills/` 目录下扫描 `*.yaml` 或 `*.md`(带 front-matter),不再是无结构文本。

### 4.3 `DAGExecutor`(新增,放 core/dag.py)

```python
class DAGExecutor:
    def __init__(self, learning: LearningAgent, orchestrator: OrchestratorAgent):
        self.learning = learning
        self.orchestrator = orchestrator

    async def execute(
        self,
        skill: Skill,
        user_input: str,
        context: Context,
    ) -> AsyncIterator[StepResult]:
        """按拓扑序执行 skill.steps:
        1. 拓扑排序(基于 depends_on)
        2. 并行执行所有 indegree=0 的 step
        3. 每完成一个,推进下游 step
        4. 失败时按 fallback / 重试策略处理
        5. yield StepResult,让上层流式推送
        """
        ...
```

### 4.4 `Context` 真正落地

[core/context.py](../core/context.py) 已定义 `Context`/`Message`,但当前 `Agent.chat()` 不使用。改进:

- 后端 `Agent` 实例**按 client_id 持久化 Context**
- 每次 `chat()` 自动 `add_user_message` / `add_assistant_message`
- Orchestrator 生成回答时,把 Context 转成 LLM messages 喂入
- 提供 `compress()` 方法,token 接近上限时丢弃/总结早期消息

### 4.5 WebSocket 协议统一(改 backend/main.py)

现状 `step` 和 `log` 两套消息结构。统一为:

```jsonc
// 所有消息统一格式
{
  "type":   "event",         // 固定
  "event":  "thinking" | "tool_call" | "tool_result"
          | "skill_step"     | "message_delta"
          | "message_final"  | "error"
          | "log",
  "trace_id": "trace-...",
  "ts":     "2026-07-07T10:30:00",
  "payload": { ... }          // 按 event 不同
}
```

并:
- `client_id` 改为 UUID4,登录时下发
- 服务端维护 `sessions: Dict[client_id, Session]`,Session 持有 Agent + Context
- WebSocket 断线后,**Context 不丢**,重连后用同一 client_id 恢复
- `try/finally` 中 unsubscribe,避免内存泄漏

## 5. 引入"教导闭环"

新增 `agents/skill_trainer.py`:

```
用户: "以后分析问题应该先收集数据,再制定方案,最后执行"
                │
                ▼

   ├─ 启发式: 包含 "以后/记住/下次/原则/方法" 等关键词

   └─ LLM 输出 {name, method, capability, patterns, tags, steps[]}

                ▼
返回: "已记住新技能: <name>"
```

触发条件:**不打断正常提问**;只有检测到明确教导意图才走这条路径。

## 9. Skill Manager Agent

> 技能库智能管理系统，负责技能检索、创建、更新和整理。

### 9.1 四大职责

| 职责 | Agent | 方法 |
|------|-------|------|
| **1. 检索技能** | `SkillRetrievalAgent` | `find_matching_skills()` - LLM 判断任务覆盖度 |
| **2. 创建技能** | `SkillCreatorAgent` | `create_skill_from_teaching()` |
| **3. 更新技能** | `SkillUpdaterAgent` | `update_skill()` - 基于反馈更新 |
| **4. 整理技能** | `SkillOrganizerAgent` | `analyze_skill_duplication()` / `merge_skills()` |

### 9.2 架构图

```
┌─────────────────────────────────────────────────────────────┐
│                    SkillManagerAgent                         │
│                       (统一入口)                             │
├───────────────┬───────────────┬─────────────┬────────────┤
│ SkillRetrieval│ SkillCreator  │ SkillUpdater│ Organizer  │
│    Agent      │    Agent     │    Agent    │   Agent    │
├───────────────┴───────────────┴─────────────┴────────────┤
│                    SkillStore                               │
│                (skills/*.yaml)                            │
└───────────────────────────────────────────────────────────┘
```

### 9.3 核心类

```python
@dataclass
class SkillMatch:
    skill: Skill           # 匹配的技能
    score: float          # 匹配度 0-1
    reason: str           # 匹配原因
    coverage: str        # "full" / "partial" / "none"

class SkillManagerAgent:
    """统一入口"""
    retrieval: SkillRetrievalAgent
    creator: SkillCreatorAgent
    updater: SkillUpdaterAgent
    organizer: SkillOrganizerAgent

    async def find_skills_for_task(user_input: str) -> Tuple[bool, List[SkillMatch], str]
    async def create_skill(user_input: str) -> Tuple[bool, str, Optional[Skill]]
    async def update_skill(skill_name: str, feedback: str) -> Tuple[bool, str, Optional[Skill]]
    async def organize_skills() -> Tuple[bool, str, List[SkillAnalysis]]
    async def merge_skills(skill1: str, skill2: str) -> Tuple[bool, str, Optional[Skill]]
```

### 9.4 技能检索流程

```
用户输入: "帮我写日报"
    │
    ▼
SkillRetrievalAgent.find_matching_skills()
    │
    ▼
构建 LLM Prompt:
  "用户需求: 帮我写日报

   请分析哪些技能能够覆盖用户需求"
    │
    ▼
LLM 返回: [
  {skill_name: "日报生成器", score: 0.95, coverage: "full"},
  {skill_name: "文本格式化", score: 0.6, coverage: "partial"}
]
    │
    ▼
返回匹配结果给用户
```

### 9.5 技能整理流程

```
定时触发 / 手动触发
    │
    ▼
SkillOrganizerAgent.analyze_skill_duplication()
```
