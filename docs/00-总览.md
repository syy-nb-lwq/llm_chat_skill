# Skill Agent 文档总览

本文档集描述的是 2026-07-21 仓库中的实际实现（M0~M4 + C-01~C-04 全部 44 项任务首轮实现完成），而不是历史设计稿。

## 项目定位

Skill Agent 是一个以 `Skill` 为中心的多 Agent 编排项目。

- `ManagerAgent` 负责意图识别、技能选择、工具规划
- `LearningAgent` 负责执行工具任务和 DAG
- `OrchestratorAgent` 负责整合结果并生成最终回复
- `SkillTrainer` 负责把用户显式教学沉淀为技能文件

系统同时提供三种入口：

- FastAPI REST + WebSocket PubSub
- Vue 3 前端
- CLI / Streamlit 同步入口

## 当前实现状态

**主执行链路（M0 完成）**

- 每次 `Agent.handle()` 生成唯一 `execution_id`，避免失败记录互相覆盖
- 工具执行记录真实名称/retry/fallback/latency_ms
- 主执行入口统一到异步 `Agent.handle()`
- 四级标识 `user_id / session_id / turn_id / execution_id`（`core/identity.py`），重试通过 `parent_execution_id` 关联父执行

**技能系统（M1 完成）**

- `TeachingSession` 多轮教学状态机持久化（Collecting→Draft→Active）
- 技能版本不可变，同名多版本并存，`name@version.yaml` 不可原地覆盖
- `active` 指针显式激活版本，`match()` 只返回 active 版本
- Skill 验证流水线（DAG 无环/工具存在/schema/样例）
- 重复技能决策：reuse / update_new / cancel
- SkillManager 接入主链，支持 MANAGER 意图（列出/版本/回滚）
- 修复 `_complete_teaching()` bug（M1-07），不再把 assistant 追问当作 user 消息回写

**长期记忆（M2 完成）**

- `MemoryRepository`（`core/memory_repository.py`）统一所有记忆读写，业务代码不再直接读写 JSON/JSONL/SQLite
- `MemoryItem` 数据模型：`tenant_id / user_id / project_id / scope / type / source_turn_id / confidence / valid_until / supersedes_id / status` 等字段
- 召回按 `tenant_id + user_id` 过滤，混合 FTS5 全文检索 + 向量相似度；按作用域/置信度/时效排序
- `MemoryStore` 旧 API 继续可用，内部委托到 Repository
- 启动时初始化 embedding service（M2-04），失败降级为全文检索不阻塞主流程
- 修复异步检索 bug（M2-05），`recall()` 改为真正的 `await`，不再被静默忽略
- Session 摘要替代无限滚动（M2-07），超出窗口的早期消息折叠为摘要注入 system prompt
- 记忆用户控制：`DELETE /api/memory/{item_id}` / `DELETE /api/memory`（forget_user）

**反馈驱动演进（M3 完成）**

- 统一 patch schema：`target_skill + version_target + diff + recommendations + evidence_execution_id`
- `FeedbackEvent`（`core/feedback.py`）绑定 `execution_id`，分类 accept/reject/correction/retry/rating
- `ResultValidator`（`core/result_validator.py`）：检查 capability 覆盖、method 步骤、examples 风格；没有工具的任务不再无条件得 100%
- Patch 审批生成新版本（M3-04），不原地改 YAML；审批时跑旧样例 + 用户纠正样例回归
- 回归失败阻止发布（M3-05），展示 diff 与 `risk_summary`
- 高置信度自动发布收紧（M3-06），`auto_approved` patch 必须经过 Validator + 回归样例 + 审计记录才落盘

**受控工具沉淀（M4 完成）**

- `ToolProposal` 数据模型（`tools/proposal.py`）+ `name@version.yaml` 不可变版本
- 声明式 HTTP 工具（`tools/declarative_http.py`）：从 proposal 渲染成 `Tool` 注册到 `ToolHub`
- 沙箱测试运行器（`SandboxRunner`）：静态校验 + 多用例执行 + `success`/`expect_error` 断言，失败不污染主进程
- Secret 引用命名空间化（如 `github.token`）+ `network_policy.allowed_hosts` 白名单（支持 `*.example.com`）+ `require_https` 强制
- `ToolApprovalService`（`tools/approval.py`）：sandbox → approve → publish → disable 全生命周期，记录审计日志
- 副作用分级 `read_only / local_write / network_write / destructive`，仅 `read_only` 可自动发布

**身份与安全（C-01 完成）**

- `owner_token` 装饰器 + `Depends(require_owner_token)` 依赖保护 13 个写管理路由
- `client_id` 服务端签发（`gen_client_id()`），不再信任客户端提交；WebSocket init 事件下发 `server_client_id`
- `get_user_from_request/WS` / `extract_ws_identity` 统一身份提取
- `owner_token` 留空时放行（单机个人环境兼容）

**ToolHub 启动可观测性（M0-06 完成）**

- `ToolHub` 维护每个源的状态机：registered / connecting / connected / connect_failed / disconnected
- `GET /api/health` 暴露 `tool_sources` 聚合 + `sources` 详情;失败时顶层 status 降级为 `degraded`
- `connect_all()` 单源失败不影响其他源，失败状态被显式记录

**配置诊断命令（C-02 完成）**

- `ui/cli.py` 重构为 argparse 双子命令：`chat`（默认）/ `diagnose`
- `diagnose` 一次性展示 provider / feature flags / 工具源 / embedding 状态 / `config.validate()` 结果，ASCII 标记跨编码可读

**Patch 审计与回滚 UI（C-03 完成）**

- 审批记录：提案来源、模型提示版本、diff、测试结果、审批人、发布时间、被替代版本
- 前端 `PatchReview.vue` / `EvolutionDashboard.vue` 提供审批与回滚操作入口

**源码编码统一（M0-05 完成）**

- 所有源码统一 UTF-8（无 BOM、LF 换行、所有 `open`/`read_text`/`write_text` 显式 `encoding="utf-8"`）
- `infra/logger` 用 `_Utf8Stream` 包装 stdout，避免 Windows cp936/cp932 下中文日志乱码
- `core/memory_db` metadata / tags 写入使用 `ensure_ascii=False`，sqlite 列里直接是原生中文而不是 `\uXXXX` 转义
- JSON-RPC 走 stdio 的 MCP 链路保留 ASCII 兼容（`ensure_ascii` 默认）

**端到端闭环（M1-10 / M2-09 / M3-07 / M4-06 完成）**

- `tests/e2e/` 提供隔离运行时夹具 + ScriptedLLMProvider，完全离线运行不依赖 OPENAI_API_KEY
- 4 个 e2e 用例覆盖 Teach → Confirm → Persist → Restart → Recall → Execute 全链路（M1-10）
- 4 个 e2e 用例覆盖重启恢复 + 用户隔离 + 删除后不再召回 + forget_user（M2-09）
- 1 个 e2e 用例覆盖用户纠正 → 新版本草稿 → 测试通过 → 切 active → 回滚（M3-07）
- 4 个 e2e 用例覆盖声明式工具 draft → sandbox → approve → publish → call → disable（M4-06）

## P10 集成补全（2026-07-22）

修复架构评审中发现的 4 处"模块存在但未接入主链路"缺口：

- **Manager 动态工具列表** — `PLAN_SCHEMA` 与 `system_prompt` 不再硬编码 `weather_query, web_search`，改为从 `ToolHub` 动态读取已注册工具名（`agents/manager.py`）
- **M4 工具提案 REST 路由** — 新增 9 个 `/api/tools/proposals/*` 端点（list/get/create/sandbox/approve/reject/publish/disable/audit），`ToolApprovalService` 全生命周期可通过 HTTP 访问（`backend/main.py`）
- **MCP 源接入启动** — `mcp_enabled=true` 时解析 `mcp_servers` JSON 并注册 `MCPSource` 到 `ToolHub`（`backend/main.py` startup）
- **Heartbeat 接入启动** — 新增 `heartbeat_enabled` / `heartbeat_interval_seconds` / `heartbeat_path` 配置，startup 启动 `HeartbeatScheduler`，shutdown 停止（`infra/config.py` + `backend/main.py`）
