## 1. M0：建立可信基线

> 目标：先证明现有系统能稳定执行和观察。

### M0-01 每个 Turn 生成唯一 `execution_id`

- 优先级：高
- 范围：`core/agent.py`、相关记录入口
- 内容：
  - 引入 `user_id / session_id / turn_id / execution_id` 四级标识
  - `Agent.handle()` 入口创建新 `execution_id`
  - 重试时建立 `parent_execution_id`
  - 修复失败 JSON 互相覆盖问题
- 来源：`10 §4.6 §6.1`
- 验收：同 Session 多 Turn 失败记录不再互相覆盖

### M0-02 补齐执行记录字段

- 优先级：高
- 范围：`core/critic.py`、`core/agent.py`
- 内容：
  - 真实工具名（不再由 task id 推断）
  - retry、fallback、错误信息、latency
- 来源：`10 §4.5`、`02 §6`、`09 §8`
- 验收：Execution 记录可还原完整执行链路

### M0-03 修复 WebSocket 集成测试

- 优先级：高
- 范围：`tests/`、测试夹具
- 内容：
  - 为 `/pubsub` 增加稳定测试夹具
  - `init / ping / chat / reset` 全链路不依赖人工
- 来源：`10 §4.10`、`08 任务2`、`06 §6`
- 验收：端到端测试不再 `skip`

### M0-04 前端 API/WS 配置化

- 优先级：中
- 范围：`frontend/`
- 内容：
  - 抽离 `API_BASE`、WS 地址
  - 支持 dev / prod 切换
- 来源：`08 任务3`、`06 §6`、`07 §1`
- 验收：代码中无 `localhost:8000` 硬编码

### M0-05 修复源码编码问题

- 优先级：中
- 范围：`backend/websocket_handler.py`、前端组件、部分中文源码
- 内容：
  - 统一 UTF-8
  - 消除乱码注释和字符串
- 来源：`08 任务1`、`00 §4`、`06 §6`
- 验收：编辑器与终端无乱码；接口行为不变
- 完成：2026-07-21（infra/logger.py `_Utf8Stream` 包装 stdout；core/memory_db.py 用 `ensure_ascii=False` 写 metadata/tags；tests/test_m0_05_source_encoding.py 10 个用例通过；全量 5 failed / 233 passed / 9 errors，失败均为预存在）

### M0-06 ToolHub 启动可观测性

- 优先级：中
- 范围：`tools/hub.py`、`backend/main.py`
- 内容：
  - 区分"源存在但连接失败"和"未注册"
  - 健康检查暴露工具源状态
- 来源：`08 任务4`、`04 §6`、`07 §5`
- 验收：`/api/health` 能列出工具源状态 ✅（2026-07-20 完成）
  - `ToolHub` 状态机：`registered` / `connecting` / `connected` / `connect_failed` / `disconnected`
  - `get_source_status()` + `health_summary()` 提供聚合
  - `/api/health` 暴露 `tool_sources` + `sources`,失败时 `status=degraded`
  - 新增 `tests/test_m0_06_tool_hub_observability.py`：13 个用例

### M0-07 增加 e2e 测试夹具

- 优先级：中
- 范围：`tests/e2e/`
- 内容：

## 8. 关键风险与不做事项（来源 `10 §13`）

- 把所有对话原文无差别写入长期记忆
- 依靠提示词解决教学/版本/权限状态
- 用更多向量库替代未打通的数据写入与身份隔离

## 9. 首个纵向用例

来源：`10 §11`

> 推荐以"日报技能"为第一个完整样板，覆盖多轮教学、用户偏好记忆、无工具技能选择、结构化输出验证、服务重启后召回、用户纠正、技能版本升级和回滚。

教学对话样例：

```text
用户：以后帮我整理日报。
系统：需要包含哪些栏目？
用户：今日完成、问题、明日计划。
系统：风格和敏感信息有什么要求？
用户：简洁，但不要太简单，不披露客户名称和内部地址。
系统：展示技能草稿、输入输出样例和约束，请用户确认。
用户：确认。
系统：验证并发布 DailyReport v1.0.0。
```

后续纠正样例：

```text
用户：问题部分如果没有内容就不要显示。
系统：将纠正绑定到上次执行，生成 v1.0.1 草稿和新增回归样例。
用户：确认更新。
系统：测试通过后切换 active 版本，保留 v1.0.0 以便回滚。
```

## 10. 验收矩阵（端到端闭环）

来源：`10 §12`

最关键的验收不是单元测试数量，而是以下端到端闭环全部可重复通过：

```text
Teach -> Confirm -> Persist -> Restart -> Recall -> Execute
      -> Correct -> Version -> Test -> Approve -> Rollback
```

| 能力 | 必须覆盖的测试 |
|---|---|
| 短期上下文 | 多轮指代、重试、Session 隔离、TTL |
| 长期记忆 | 写入、重启恢复、用户隔离、冲突、删除、过期 |
| 技能教学 | 信息不足追问、草稿修改、取消、确认发布 |
| 技能检索 | 纯方法论技能、工具技能、低置信度澄清、禁用版本 |
| 技能执行 | 输入校验、DAG、retry、fallback、超时、输出校验 |
| 技能演进 | 用户纠正、新版本、回归失败阻止发布、回滚 |
| 工具治理 | schema、权限、沙箱、secret、审批、禁用 |
| API/WebSocket | 身份、重连、事件顺序、错误恢复、未授权访问 |
| 并发与部署 | 同用户多 Session、多用户、进程重启、多实例一致性 |
